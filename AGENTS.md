# tasks.nvim — AI Coding Assistant Guide

A task manager for Neovim, inspired by VSCode's tasks system. Manages task definitions
in TOML config files, supports variable expansion, custom providers, and integrates with
telescope.nvim / picker.nvim for task selection.

## Coding Conventions

### Lua Style

- Formatting: [StyLua](https://github.com/JohnnyMorganz/StyLua) with config in `.stylua.toml`
- Indentation: 2 spaces
- Line length: 120 characters max
- Quotes: Single quotes for strings
- Naming: `snake_case` for functions and variables, `PascalCase` not used (no classes)

### Module Pattern

```lua
local M = {}

-- Private variables (module-local)
local state = {}

-- Public functions (exposed on M)
function M.public_function()
  -- ...
end

return M
```

### File Headers

All source files in `lua/tasks/` use the standard header:

```lua
--=============================================================================
-- filename.lua
-- Copyright (c) 2016-2022 Wang Shidong & Contributors
-- Author: Wang Shidong < wsdjeg@outlook.com >
-- License: GPLv3
--=============================================================================
```

### Variable Expansion

Task commands and args support VSCode-style variables:

| Variable | Expands To |
|----------|------------|
| `${workspaceFolder}` | Project root directory |
| `${workspaceFolderBasename}` | Project root directory name |
| `${file}` | Current file full path |
| `${relativeFile}` | Current file relative path |
| `${fileBasename}` | Current file name |
| `${fileDirname}` | Current file directory |
| `${fileExtname}` | Current file extension |
| `${lineNumber}` | Current cursor line |

## Testing

Framework: **luaunit**. Test files: `test/*_spec.lua`.

### Running Tests

```bash
# Run all tests
make test

# Run specific tests by pattern (matches file names)
make test PATTERN=example

# Run by full path
make test PATTERN=test/example_spec.lua

# Install test dependencies only
make install-deps

# Clean test artifacts
make clean
```

### Writing Tests

```lua
local lu = require('luaunit')
local tasks = require('tasks')

TestMyFeature = {}

function TestMyFeature:test_something()
  lu.assertEquals(1 + 1, 2)
end

-- Run the suite
os.exit(lu.LuaUnit.run())
```

### Test Environment

`test/minimal_init.lua` sets up:
- Package paths for `lua/`, `test/`, `test/.deps/`
- Temporary test directory with sample TOML files
- Plugin initialization via `require('tasks').setup()`
- Global variables (`_G.TEST_DIR`, `_G.GLOBAL_TOML`, `_G.LOCAL_TOML`) for test access

### CI

Tests run on push to `main`/`master` and on PRs, across:
- **OS:** ubuntu-latest, windows-latest, macos-latest
- **Neovim:** stable, nightly

## Commit Style

Follow [Conventional Commits](https://www.conventionalcommits.org/). Format: `type(scope): subject`

| Type | For | Triggers Release |
|------|-----|------------------|
| `feat` | New feature | Minor bump |
| `fix` | Bug fix | Patch bump |
| `refactor` | Code restructure | None* |
| `docs` | Documentation | None |
| `test` | Tests | None |
| `ci` | CI/CD changes | None |
| `chore` | Maintenance | None |
| `perf` | Performance | Patch bump |
| `style` | Formatting | None |
| `build` | Build system | None |
| `security` | Security fix | Patch bump |

\* Unless `BREAKING CHANGE` footer or `!` after type (e.g., `refactor!: change API`).

**Rules:**
- Imperative mood: "add feature" not "added feature"
- Lowercase, no trailing period
- Subject line under 72 characters
- Scope is optional: `fix(provider): handle missing npm`

**Examples:**
```
feat: add cargo task provider
fix(utils): correct path separator on Windows
docs: update README with telescope setup
test: add expand_task variable expansion tests
ci: add macOS to test matrix
```

## Release Process

Releases are automated via [release-please](https://github.com/googleapis/release-please).

### How It Works

1. Commits pushed to `master` trigger the `release-please.yml` workflow
2. release-please analyzes commit messages and creates/updates a release PR
3. The release PR bumps version in `.github/release-please-manifest.json` and updates `CHANGELOG.md`
4. When the release PR is merged, release-please creates a git tag and GitHub Release
5. The tag push triggers `luarocks.yml` to publish to LuaRocks

### Manual Release PR Management

To re-trigger or fix the release PR:

```bash
git fetch origin
git checkout release-please--branches--master
git reset --hard origin/release-please--branches--master
git rebase master
git push --force origin release-please--branches--master
git checkout master
git merge release-please--branches--master
```

### Do NOT Manually Create Tags

Release-please automatically creates git tags and GitHub Releases when the release PR is merged. Manual tag creation will conflict with the automation and cause version mismatches.

## Key APIs

### Module: `tasks`

```lua
local tasks = require('tasks')
```

| Function | Description |
|----------|-------------|
| `setup(opt)` | Initialize plugin with config |
| `get()` | Load tasks, show picker menu, return selected task |
| `list()` | Load tasks, show task viewer buffer |
| `edit(global)` | Edit global (`true`) or local (`false`) tasks file |
| `get_tasks()` | Load and return all tasks as a table |
| `expand_task(task)` | Expand variables in a task definition |
| `reg_provider(fn)` | Register a custom task provider function |

### Configuration

```lua
require('tasks').setup({
  global_tasks = '~/.tasks.toml',  -- Global tasks file path
  local_tasks = '.tasks.toml',     -- Project-local tasks file path
})
```

### Task Definition Format (TOML)

```toml
[task-name]
command = "make build"
description = "Build the project"
args = ["--verbose"]
isGlobal = false

[task-name.options]
cwd = "${workspaceFolder}"
env = { KEY = "value" }

# OS-specific overrides
[task-name.windows]
command = "make build.exe"

[task-name.osx]
command = "make build-osx"
```

### Custom Provider

```lua
tasks.reg_provider(function()
  return {
    ['detected-task'] = {
      command = 'npm test',
      description = 'Run npm test',
      isDetected = true,
      detectedName = 'npm: ',
    }
  }
end)
```

3. Run `make test` — all tests must pass
